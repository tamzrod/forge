# Deploy Directory

This directory contains everything required to deploy, maintain, reset, and operate the application.

## Purpose

The deploy directory standardizes deployment operations across all ecosystem projects:
- MMA2
- Atlas-PPC
- Simulation Runtime
- Librarian
- Future projects

Developers moving between projects immediately recognize the structure and operational interface.

## Directory Structure

```
deploy/
├── README.md           # This file
├── docker/            # Docker and container configuration
├── config/            # Application configuration
├── data/              # Runtime-generated data (gitignored)
├── logs/              # Log files (gitignored)
├── backup/            # Backup storage (gitignored)
└── scripts/           # Operational scripts
```

### docker/

Container definitions:
- Dockerfile
- docker-compose.yml
- .dockerignore

### config/

Application configuration:
- runtime.yaml
- plugins/
- protocols/

### data/

Runtime-generated data, gitignored:
- databases
- caches
- temporary files

### logs/

Application logs, gitignored:
- runtime.log
- scheduler.log

### backup/

Backup storage, gitignored:
- Backups created by ./scripts/backup.sh

### scripts/

Operational scripts - see Script Reference below.

## Script Reference

All scripts are executable. Run from project root.

```bash
./deploy/scripts/up.sh        # Start the application
./deploy/scripts/down.sh      # Stop the application
./deploy/scripts/restart.sh   # Restart the application
./deploy/scripts/build.sh     # Build without starting
./deploy/scripts/rebuild.sh  # Rebuild and restart
./deploy/scripts/reset.sh    # Factory reset (preserve config)
./deploy/scripts/nuke.sh     # Destroy everything
./deploy/scripts/logs.sh     # Show logs
./deploy/scripts/shell.sh    # Shell into runtime
./deploy/scripts/health.sh   # Health check
./deploy/scripts/backup.sh   # Backup data
./deploy/scripts/restore.sh  # Restore data
```

## Script Definitions

### up.sh

Starts the application. Idempotent if already running.

### down.sh

Stops the application cleanly. Waits for graceful shutdown.

### restart.sh

Combines down + up with brief pause between.

### build.sh

Builds images and binaries without starting services.
Use after code changes without restarting.

### rebuild.sh

Performs clean rebuild:
1. Stop services
2. Remove containers/volumes
3. Build images
4. Start services

### reset.sh

Factory reset. Removes runtime-generated data while preserving configuration:
- databases
- caches
- temporary files
- logs

Does NOT remove:
- Configuration files
- Docker images
- Source code

### nuke.sh

Destroys everything generated by the project:
- containers
- images
- volumes
- databases
- caches
- generated files

**Requires explicit confirmation.** Type "nuke" when prompted.

### logs.sh

Displays runtime logs. Supports:
- All logs: `./deploy/scripts/logs.sh`
- Follow mode: `./deploy/scripts/logs.sh -f`
- Filter: `./deploy/scripts/logs.sh --grep error`

### shell.sh

Opens a shell inside the primary runtime container.
Fails gracefully if not running in container.

### health.sh

Performs health checks:
- Runtime process running
- Scheduler active
- Plugins loaded
- Protocols available

Exit codes:
- 0: Healthy
- 1: Unhealthy
- 2: Unknown state

### backup.sh

Creates timestamped backup of persistent data:
- Databases
- Configuration snapshots
- Runtime state

Backups stored in `deploy/backup/`.

### restore.sh

Restores from backup. Prompts for backup selection.

## Future Extensions

The structure supports future deployment targets:

### Docker Compose

Current implementation uses docker-compose.yml in `docker/`.

```

├── deployment.yaml

├── forge.timer
```

### Windows Service

Future addition:
```
deploy/windows/
├── forge.exe
├── install.ps1
```

## Design Principles

1. **Consistency** - Same structure across all projects
2. **Simplicity** - Scripts are small, readable, documented
3. **Idempotency** - Scripts can be run multiple times safely
4. **Fail Gracefully** - Clear error messages, no surprises
5. **No Magic** - Scripts should be understandable at a glance

## Extending Scripts

When adding new scripts:

1. Place in `./deploy/scripts/`
2. Make executable: `chmod +x script.sh`
3. Document in this README
4. Use consistent header:

```bash
#!/bin/bash
# script.sh - Brief description
# Usage: ./deploy/scripts/script.sh [args]
set -e

SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
# ...
```
